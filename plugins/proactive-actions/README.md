# proactive-actions

> Scan action items from meeting debriefs, classify what can be handled autonomously, and execute without waiting to be asked.
> v0.1.0 — `medium` tier.

## What it does

After meeting debriefs are processed, scans all action items and commitments, classifies each by what the agent can handle autonomously, and acts:

- **Executes immediately** what the agent can handle without human involvement — creating task board entries, drafting documents, writing guides, updating project files.
- **Drafts what needs review** — scripts, emails, frameworks, or training content that requires the user's voice or approval. Flags for review, never sends.
- **Chases outstanding commitments** — when someone else committed to something, sends a status-check message to confirm it's in progress or surfaces it if overdue.
- **Escalates what needs a human** — adds to the task board with full context, surfaces in the next start-of-day briefing.

The result: after every meeting batch, the user's task board is updated, follow-ups are sent, and everything that can be handled is already handled.

## Who it's for

Operators and executive assistants running a high volume of meetings with a lot of resulting action items. Atlas built this for EA workflows where the bottleneck isn't identifying what needs to happen — it's executing the routine parts fast enough that the human's attention stays on high-judgment work.

## Required capabilities

The plugin's skills depend on these capabilities. Each is named abstractly — wire it up to whatever tools the host agent has access to.

- **Meeting debrief read** — read structured debrief files generated by the `meeting-ops` plugin (or any equivalent debrief format)
- **Task system write** — create and update tasks in the user's task board or to-do system
- **Messaging send** — send internal follow-up messages to team members (internal only — never external)
- **File write** — write draft documents, guides, and frameworks to the knowledge base
- **Rules file read** — read `AGENTS.md` for permission rules before taking any autonomous action

## Suggested tool wiring

| Capability | Common options |
|---|---|
| Meeting debrief read | Filesystem MCP (markdown files), Notion MCP, any meeting log store |
| Task system write | Linear MCP, Todoist, Things, a markdown task file |
| Messaging send | Slack MCP, Teams MCP, any internal messaging tool |
| File write | Filesystem MCP, Notion MCP |
| Rules file read | Filesystem MCP |

These are examples, not requirements. Pick what the agent already has.

## Installation

```
/plugin install proactive-actions@atlas
```

After installing, complete the first-run setup below.

## First-run setup

> See [`instructions.md`](instructions.md) for the full setup walkthrough.

1. **Meeting debrief location.** Where debrief files live — path and filename convention. Default: `brain/meetings/debriefs/YYYY-MM-DD-*.md`. Works out of the box with the `meeting-ops` plugin.
2. **Task system wiring.** Connect the task system capability to the user's actual tool. Configure owner labels for the agent vs. the user.
3. **Messaging capability and channel routing.** Wire up the messaging tool. Configure which workspace or channel to use for internal follow-ups vs. status reports to the user.
4. **`AGENTS.md` path.** Confirm the agent's rules file is readable. The skill checks this before every autonomous action.
5. **Escalation channel.** Where to send the end-of-run status report to the user. Default: the user's primary DM or operations channel.

## Skills included

- **`run-proactive-actions`** — *opinionated.* Scans today's debriefs, classifies every action item into one of four buckets (CAN EXECUTE / CAN DRAFT / CAN CHASE / NEEDS HUMAN), and processes each one. Produces a structured status report.

## Customization notes

Common things clients change:

- **Classification rules.** The four-bucket classification logic lives in `references/atlas-proactive-actions-methodology.md`. Override which action types land in which bucket.
- **Chase timing.** Default is: don't chase same-day commitments unless marked ASAP. Override the delay threshold in `skills/run-proactive-actions/SKILL.md`.
- **Escalation report format.** The default status report format is in the methodology reference. Edit it to match how the user wants to receive the summary.
- **Task ownership labels.** Default uses `atlas` for agent-owned tasks and the user's name for human-owned tasks. Change to match your task system's label conventions.
- **Messaging scope.** This skill sends messages only to internal team by default. Customize the approved contact list if your "internal" scope differs.

When customizing, edit the `SKILL.md` and reference files in your installed copy or fork.

## Atlas methodology

This plugin encodes Atlas's four-bucket classification framework — the discipline that prevents two failure modes: doing too little (leaving everything on the user) and doing too much (taking autonomous actions that needed approval).

- **CAN EXECUTE** — do it immediately, log it, update the task board.
- **CAN DRAFT** — write it, flag it for review, never publish it.
- **CAN CHASE** — send a friendly status check, not a nag. Don't chase same-day.
- **NEEDS HUMAN** — add to task board with full context. Don't attempt; don't skip; don't block.

The full framework lives at [`references/atlas-proactive-actions-methodology.md`](references/atlas-proactive-actions-methodology.md).

## Troubleshooting

**Skill says no debriefs found.** Check the debrief path and date. Default looks for today's debriefs — if you're running this the morning after, adjust the date range or pass the path explicitly.

**Chase message went to the wrong workspace.** Check the channel routing configuration in first-run setup. Internal team messages and user status reports route separately.

**An action was classified in the wrong bucket.** Override the classification logic in the methodology reference. The four-bucket rules are editable.

**Task board entries are missing owner labels.** Check that the task system is configured with the correct owner labels for agent vs. human tasks.

**The status report didn't reach the user.** Confirm the escalation channel in first-run setup. If the messaging capability isn't wired to the user's DM or ops channel, the report is silently lost.
